## Goal

Find and fix outright incorrectness. Result: the render/format engine is **robust**. Across 11
targeted reproductions (rounding, encoding/CJK, None handling, wrap-conflict modes, colspan
validation, the `max_width` invariant, streaming late-row behavior, strict-combine data
integrity) there were NO crashes, NO data-loss paths, and NO correctness failures on reachable
paths. Three minor, low-risk nits remain, proposed below.

## What was verified correct (NOT changed; recorded so it is not re-litigated)

- `max_width` is never exceeded across plain / colspan / unbreakable-token / CJK stress cases,
  and every rendered line is width-uniform.
- CJK 2-wide chars align correctly at narrow `max_width` (11-15) with no off-by-one.
- All `on_wrap_conflict` modes (warn/clip/overflow/error) behave as intended.
- `set_cell_span` validation raises clear, specific errors for span-past-end / col- / row-
  out-of-range; `combine=None` strict mode blocks silent overwrite of a non-empty covered cell.
- Streaming holds fixed geometry from the sample (a late wide row is fit to it, not misaligned) -
  the documented tradeoff, not a bug.
- No bare `except:`, no mutable default arguments, no obvious swapped-arg/off-by-one in the
  width/wrap core.

## Findings

Severity = impact if left alone; Remediation Risk = Fix-Bar gate.

| ID | Severity | Remediation Risk | Persona | Area | Finding | Evidence |
|----|----------|------------------|---------|------|---------|----------|
| B1 | Medium | Low | QA | integration-correctness / surprising output | Integer formatting uses Python `round()`, which is round-half-to-EVEN (banker's). So `i`/`I` render `0.5 -> 0`, `2.5 -> 2`, `3.5 -> 4`. A user expecting round-half-up gets a silently "wrong-looking" integer, and the behavior is undocumented. Reachable via any `.5` value in an int column. | src/vistab.py:2506 (`str(int(round(...)))`), :2514 (`f"{int(round(...)):,d}"`); reproduced: i(2.5)=2, i(0.5)=0 |
| B2 | Low | Low | engineer | swallowed error | A malformed/invalid `~/.config/vistab/themes.json` is swallowed by `except Exception: pass`, so custom themes silently fail to load with no warning; the user cannot tell their config is broken. | src/vistab.py:3762-3766 |
| B3 | Low | Low | QA | contract / None handling | A `None` cell in a numeric (`i`/`f`/`a`) column renders the literal string `"None"` rather than a blank or a clear signal. It does not crash (FallbackToText), but `"None"` in a number column is misleading output. | reproduced: `i <- None` -> `"None"`; FallbackToText path in `_str`/`_fmt_*` |

No Blocker/High findings. No data-loss, corruption, auth, or crash-on-reachable-path defects.

## Proposed changes (ordered, validatable)

| Step | Source finding IDs | Change | Files | Remediation Risk | Validation |
|------|--------------------|--------|-------|------------------|------------|
| 1 | B1 | Decide and document the integer rounding policy. Preferred: switch `_fmt_int`/`_fmt_comma_int` to round-half-up (e.g. `math.floor(x + 0.5)` for x>=0 with symmetric handling for negatives, or `decimal.Decimal(...).quantize(ROUND_HALF_UP)`), which matches common user expectation; OR, if banker's is intended, document it explicitly in API.md/docstrings. Do NOT change silently. | src/vistab.py:2506,2514; docs/API.md | Low | New tests pin the chosen behavior for {0.5,1.5,2.5,-0.5,-1.5}; docstring/API updated; other fixtures unchanged. |
| 2 | B2 | On a themes.json that exists but fails to parse, emit a one-line WARNING to stderr (path + reason) instead of silent `pass`; still continue without the custom themes. | src/vistab.py:3762-3766 | Low | A deliberately-corrupt themes.json produces a stderr warning naming the file; a valid one still loads silently; startup never crashes. |
| 3 | B3 | Render `None` in a numeric column as an empty cell (or a configurable placeholder), not the literal `"None"`; keep text columns rendering `str(None)` only if explicitly text. Confirm this does not change how legitimate string "None" data renders. | src/vistab.py `_fmt_*`/`_str` None handling | Low | Test: `None` in i/f/a column -> empty (or placeholder); literal string "None" in a text column -> unchanged; no crash. |

## Open questions

- B1 is the load-bearing decision: round-half-up (match expectation; a behavior CHANGE to
  existing `.5` output) vs. keep banker's and just document it (no output change)? Recommend
  round-half-up for least surprise, but it is a shipped-behavior change, so it needs the human's
  call. This is exactly what `plan-review` / approval should decide.
- B3: empty cell vs. a configurable null placeholder (e.g. a `set_null_str("")`-style option)?
  Recommend the simplest (empty) unless a placeholder is wanted; adding an option is scope creep
  unless requested.
